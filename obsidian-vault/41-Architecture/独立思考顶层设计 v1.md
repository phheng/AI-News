## 一、先给结论（审查后）

### 1) 现有方向总体正确，但需要三处收紧
1. **指标阈值需要分层**：不能一刀切给所有客户；要按 Starter/Growth/Scale 分层阈值。
2. **Token 成本要进入经营看板主屏**：不是技术指标，而是产品毛利核心变量。
3. **“自动执行”默认边界要更严**：先 Observe/Approve，Auto 只允许低风险操作。

### 2) OpenClaw 架构仍是当前阶段最优解（局部最优）
- 快速验证产品市场匹配（PMF）
- 成本和迭代速度可控
- 与现有 cron + Obsidian + IM 工作流直接兼容

## 四、修正后的架构要点（可落地）

## 4.1 技术与运营一体化架构
- 编排层：OpenClaw（agent/workspace/cron/subagent）
- 程序层：采集->清洗->去重->聚类->评分（低 token）
- 智能层：LLM 只做解释、优先级与建议
- 交付层：IM 消息 + Obsidian 报告 + 审批队列

## 4.2 指标体系（最终口径 v1）

### A. 业务价值
- ACOS 相对改善率（%）
- ROAS 相对改善率（%）
- 断货 SKU 数变化
- 每周节省运营小时

### B. 产品运营
- DAU/WAU（租户活跃）
- 建议采纳率
- 7/30 天留存
- 试点转付费率

### C. 稳定性
- cron 任务成功率
- P50/P95 执行时长
- 报告完整率（含证据链接）

### D. 成本（重点）
- token_in / token_out / token_total
- 每次 run 成本（$/run）
- 每租户月成本（$/tenant/month）
- Token 成本占订阅收入比例（%）

## 4.3 关键告警阈值（初始）
- 任务成功率 < 95%：告警
- 报告完整率 < 98%：告警
- Token 成本占月费 > 25%：告警并触发降本策略
- 建议采纳率连续 2 周 < 20%：触发产品可用性复盘

## 五、执行白名单（风险控制）

### Auto（可自动）
- 生成日报
- 低风险提醒
- 非破坏性数据同步

### Approve（默认）
- 广告预算调整
- 关键词否词批量变更
- 补货建议落地

### Human-only（人工）
- 大额预算策略变更
- 定价策略重大调整
- 跨市场策略切换

## 六、新增：AI 搜索优化如何纳入产品设计（v1补充）

### 6.1 设计目标
- 提高“被 AI 检索/引用/推荐”的概率（而不仅是传统 SEO）
- 让 Agent 输出可被机器稳定解析、可信引用、持续更新

### 6.2 产品设计落点
1. **结构化输出优先**
- 每份关键文档固定字段：`title/date/version/source_links/confidence/action_items`
- 事件型内容强制包含证据链接与时间戳

2. **知识单元化（Chunk-first）**
- 将长文切成“问题-结论-证据”最小单元
- 每个单元可独立被检索（便于 RAG 和 AI 引用）

3. **来源可信度标签**
- 每条结论附 `source_tier(A/B/C)` 与 `last_verified_at`
- 无链接结论默认降权或标记低置信

4. **更新新鲜度机制**
- 关键页面（ICP/架构/指标）增加 `updated_bjt` 与变更摘要
- 定时任务优先更新高价值页面，避免陈旧内容被检索到

5. **面向 Agent 的可调用接口**
- 统一输出 JSON/Markdown 双格式
- 提供稳定字段名，减少模型解析偏差

### 6.3 运营指标（AI Search）
- AI 引用命中率（被内部/外部 Agent 引用次数）
- 可解析率（结构化字段完整率）
- 证据覆盖率（含来源链接结论占比）
- 新鲜度达标率（关键文档在 SLA 时间内更新）

### 6.4 风险与防偏差
- 风险：为了“可检索”牺牲可读性
- 对策：双轨输出（机器结构化 + 人类可读摘要）
- 风险：低质量来源污染知识库
- 对策：来源分级 + 定期复核 + 自动降权

## 八、结论（最终）
- 103-A 当前框架方向正确。
- 真正的成败点不是“功能多”，而是：
  1) 指标口径稳定，
  2) Token 成本受控，
  3) 高风险动作可审计可回滚。
- 在这三点被落实前，不建议扩大 Auto 执行范围。
